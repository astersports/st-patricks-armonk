1. ONE NEW CONSTRAINT THAT CHANGES THE FRAME (read before the items)
--------------------------------------------------------------------------------
Since the audit was written, the operating model was made explicit and committed
(CLAUDE.md + docs/SANDBOX_AND_MIGRATION.md, PR #14):

  - This repo is the SANDBOX. It is not production and never hosts the live site.
  - Production lives in a SEPARATE repo on a SEPARATE GitHub account (Archdiocese
    of New York organizational-separation requirement). That repo owns hosting,
    domain/DNS, SECRETS, real-PII retention, and go-live.
  - Work migrates one way, sandbox → production, via an operator-built agent that
    reads this repo. Rule of thumb: carry CODE + INTENT; never carry secrets,
    instance handles, or host wiring.

Consequence for Phase 2: both remaining items (A-5, A-7) need things that are
ENVIRONMENT-SPECIFIC and live on the production side — a Google service-account
credential (A-5) and a registered cron caller + cron secret (A-7). So the real
question is not just "build it?" but "build WHERE, and in what shape so it
migrates cleanly?"


2. DECISION D1 (decide first) — sandbox-build-now vs. rehost/production-spec
--------------------------------------------------------------------------------
REQUEST: What posture do we take for the remaining Phase-2 automation — build it
on the legacy sandbox stack now, or spec it for the Supabase/production rehost?

CONTEXT / EVIDENCE:
  - Legacy sandbox stack is MySQL/TiDB + S3 (README "Stack"); the forms-rehost
    spec re-platforms to Supabase Postgres + Storage on the production side.
  - The credentialed/infra parts of A-5/A-7 cannot be truly "finished" in the
    sandbox because the credentials and cron live in production (CLAUDE.md §2,
    docs/SANDBOX_AND_MIGRATION.md "ENVIRONMENT-SPECIFIC").
  - But the BUSINESS LOGIC (what to export, which rows are stale, what the digest
    says) is pure and migrates cleanly — exactly how A-3 was built (pure helper
    in client/src/lib/ccdReports.ts).

OPTIONS:
  (A) Build fully on the sandbox now (incl. provisioning sandbox Google creds and
      a sandbox cron). + value sooner; − partly throwaway, needs sandbox secrets,
      duplicates infra that production must re-provision anyway.
  (B) Spec-only; build nothing until the rehost. + no throwaway; − slower, no
      value to staff in the interim.
  (C) HYBRID: build the PURE LOGIC now (testable, no secrets — migrates verbatim),
      leave the credentialed/cron wiring as a documented seam for production.

CC LEAN: (C) HYBRID. It matches how A-3 already shipped, gives staff value via the
pure pieces, and produces exactly the artifact the migration agent wants. The
secret/cron wiring is prepared (interfaces + a stub endpoint) but PERFORMED in
production.


3. DECISION D2 — A-5: Google Sheets auto-export
--------------------------------------------------------------------------------
REQUEST: Do we wire scheduled auto-export of form submissions to a Google Sheet,
and in what shape?

CONTEXT / EVIDENCE:
  - A scaffold already exists: server/routers/formExport.ts —
      * getConfig/setConfig persist a spreadsheet id (site setting
        "form_export_spreadsheet_id");
      * getRecentSubmissions + exportCsv return submission rows / CSV.
    BUT the file header says the actual push uses "the GWS CLI via child_process
    (sandbox only)" — i.e. there is NO production Sheets-API path wired today;
    export is effectively manual CSV.
  - Scheduler mechanism = HTTP endpoints POST /api/scheduled/* gated by a cron
    secret, hit by an external cron caller (server/_core/index.ts:54–90:
    /api/scheduled/ccd-reminders, /api/scheduled/analytics-digest).
  - NOTE the audit's own finding O-4: the admin "NeedsAttention" inbox is already
    the staff system of record. A Google Sheet may duplicate it.

OPTIONS:
  (1) One-way scheduled push (app → Sheet), incremental (only new/changed rows).
      REQUIRES: a Google service-account credential (PRODUCTION secret), a
      /api/scheduled/sheets-export endpoint + handler, an incremental cursor, and
      a column mapping per form type.
  (2) Bidirectional sync (Sheet ↔ app, staff edits status in the Sheet).
      REQUIRES: all of (1) plus conflict handling + write-back; creates a second
      source of truth against the admin inbox (audit O-4 risk). Not recommended.
  (3) Drop A-5. The admin inbox + the per-report CSV exports (A-3 pattern) may
      already cover the real need; revisit only if staff specifically want a Sheet.

CC LEAN: (1) one-way push, but BUILT IN PRODUCTION (the service-account secret
lives there). In the sandbox now: finalize the pure per-form column-mapping as a
testable helper (mirrors ccdReports.ts) so the push handler is a thin wrapper in
production. Architect to confirm: is a Sheet actually wanted over the admin inbox,
and one-way vs bidirectional?

OPEN QUESTIONS FOR ARCHITECT:
  - Q2a: Sheet wanted at all, or is the admin inbox + on-demand CSV sufficient?
  - Q2b: If wanted — one-way (lean) or bidirectional?
  - Q2c: One tab per form type, or a unified tab? (affects the column mapping)


4. DECISION D3 — A-7: stale-submission reminders
--------------------------------------------------------------------------------
REQUEST: Add a recurring reminder so submissions don't die unread — and in what
form (staff digest vs. parishioner-facing escalation)?

CONTEXT / EVIDENCE:
  - The scheduler pattern exists and is proven (server/_core/index.ts:54–90;
    server/scheduledDigest.ts handleWeeklyDigest; server/scheduledAnalytics.ts
    handleAnalyticsDigest). Adding one more /api/scheduled/* handler is low-risk.
  - Routing by form-type → team alias already exists (server/notifications/route.ts;
    routeNotification used by every form router), so "send the digest to the right
    team" is already solved.
  - Every form table carries a status with a "pending" state and createdAt — the
    "stale" query is straightforward.

OPTIONS:
  (1) STAFF DIGEST: a weekly "these are pending > N days" summary to each team
      (sacraments → office, CCD → religious-ed, etc.). Internal only; no
      parishioner contact. REQUIRES: N threshold, a query across the form tables,
      a compose step, a /api/scheduled/stale-submissions handler + cron entry.
  (2) PARISHIONER ESCALATION: auto-email the family at 3/7/14 days ("we have your
      request; next step is…"). More aggressive; parishioner-facing; needs careful
      copy and a per-form "next step." Higher value, higher blast radius.
  (3) AUTO-DECLINE after N days (e.g. close a baptism with no cert). NOT
      recommended for a parish — pastorally wrong to auto-reject.

CC LEAN: (1) staff digest, threshold ~7 days, internal only, reusing the existing
team routing. Build the stale-query + digest-compose as PURE, TESTED logic in the
sandbox now; wire the cron registration + secret in production (D1 hybrid). Revisit
(2) parishioner escalation later if Frank wants it — it’s a separate, copy-heavy
decision.

OPEN QUESTIONS FOR ARCHITECT:
  - Q3a: Threshold — 7 days? different per form (funerals are urgent)?
  - Q3b: Staff-only (lean) or also gentle family nudges (option 2)?
  - Q3c: Cadence — weekly digest, or daily for anything > threshold?
