────────────────────────────────────────────────────────────────────────
1. THE JOB (Frank's words)
────────────────────────────────────────────────────────────────────────
Blank forms get FILLED OUT and DISTRIBUTED to the right destination so AN ACTION
HAPPENS. A parishioner completes a form on the site → it generates a PDF → the PDF is
emailed to the form's configured recipient → for the sensitive subset, the submission +
any uploaded file are also stored securely so nothing is lost and staff can act on it.

────────────────────────────────────────────────────────────────────────
2. STACK (ruled — standalone, #103 pattern; NOT a youth-sports tenant)
────────────────────────────────────────────────────────────────────────
Q1/S2 ruled: STANDALONE rehost. A parish CMS + form system is a different domain from
the youth-sports multi-tenant schema; forcing it in is a bad marriage. Reuse the #103
PATTERN, not its instance:
  - Host: Railway. · DB + Storage: Supabase (Postgres + Storage). · Staff auth: Google
    OAuth (own-JWT session, the #103 option-(b) shape). · Email delivery: a real email
    service (Resend or SendGrid) — pick one, keyless-fallback to log in dev.
  - All outbound (email API, any geocode) through the app's fetch boundary; secrets via
    env, never committed.

────────────────────────────────────────────────────────────────────────
3. FORM MODEL — a registry, not hardcoded pages
────────────────────────────────────────────────────────────────────────
Each form is a config row, so adding/editing a form is data, not code:
  form: { slug, title, intro, fields[], recipients[], sensitivity, confirmationCopy }
  field: { key, label, type (text|date|select|checkbox|file|…), required, help }
  recipients: where the filled form is routed (the "distributed somewhere" core) —
    one or more parish roles/addresses, e.g. CCD Coordinator, Parish Office, the priest.
  sensitivity: T0 (low) | T1 (sensitive — child data / cert uploads).
Initial forms (blank): CCD / Faith-Formation registration (T1), sacrament request (T1
  if cert upload), Mass intention (T0), bulletin/Flocknote signup (T0), general
  contact (T0). Exact set from the parish; the registry makes it editable.

────────────────────────────────────────────────────────────────────────
4. SUBMIT PIPELINE — fill → PDF → route → (store)
────────────────────────────────────────────────────────────────────────
On submit (public, unauthenticated):
  1. Validate server-side (required, types, file magic-byte check — the audit flagged
     St Patrick's missing upload validation; do NOT repeat it).
  2. Render a PDF of the submission (parish letterhead).
  3. ROUTE: email the PDF to the form's configured recipient(s). This is the action
     trigger. Active-voice confirmation to the submitter ("Your registration was sent
     to the Faith Formation office.").
  4. SENSITIVITY-TIERED PERSISTENCE:
     - T0: email-PDF only. No stored record needed (mass intention, contact, signup).
     - T1: ALSO write the submission row + any uploaded file to Supabase. Files →
       Supabase Storage (encrypted at rest), staff-read-only via RLS. Email a SECURE
       STAFF LINK to the file, do NOT attach a child's birth cert to a plaintext email.
  STORE-THEN-SEND for T1: persist first, then email — so a bounced email is never a
  lost registration. Log + flag email failures for staff retry.

────────────────────────────────────────────────────────────────────────
5. STAFF INBOX — so the action actually happens (light v1)
────────────────────────────────────────────────────────────────────────
Behind Google staff login: a submissions list for T1 forms — newest first, routed
recipient shown, status New | Actioned, view-PDF, view-secured-file, mark-actioned.
Purpose: a registration can't die unread in someone's inbox. RLS: only staff of the
parish read submissions; the public can submit, never read. Keep v1 minimal (list +
status + PDF/file view); status workflow can grow later.

────────────────────────────────────────────────────────────────────────
6. PRIVACY / SAFETY  (forms collect minors' data — design it in, don't bolt on)
────────────────────────────────────────────────────────────────────────
The moment a blank form goes live and is filled, you are collecting real PII again —
including children's (CCD registration, sacrament records, cert uploads). So:
  - Submissions are NEVER public. Public can POST; only staff READ (RLS). No public read
    path to any submission or file.
  - Child-data uploads (cert, etc.): Supabase Storage, encrypted at rest, staff-RLS,
    delivered to staff as a secured link — never as a plaintext email attachment.
  - Minimize: collect only the fields the action needs. No field that isn't acted on.
  - Retention: add a stated retention/disposal answer per T1 form (e.g. purge after the
    sacrament cycle). Stub now; parish confirms the period. [DECISION — parish]
  - Transport: email service over TLS; for T1, the email carries a link, not the PII
    payload, wherever feasible.

────────────────────────────────────────────────────────────────────────
7. A11Y + QUALITY (the audit flagged St Patrick weakest here — clear the bar)
────────────────────────────────────────────────────────────────────────
Labels on every field, visible focus, real error feedback (say what's wrong + how to
fix, in the interface's voice), 44px targets, reduced-motion, date inputs that are
timezone-safe (T1 finding). Forms must be fully keyboard + screen-reader usable.
